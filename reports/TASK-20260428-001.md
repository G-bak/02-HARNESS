## 1. 요청 및 목표

사용자는 Claude Code CLI 옵션을 한국어로 정리하고, Generator가 실제로 파일을 수정할 수 있는지 확인한 뒤, 실행 가능한 Generator 파이프라인까지 구축하라고 지시했다.

작업 중 추가로 다음 운영 질문이 확정됐다.

- Generator는 fresh Claude CLI 실행으로 호출한다.
- 기본 모델은 `best`, effort는 `xhigh`, permission mode는 `auto`로 둔다.
- `dangerously-skip-permissions`와 `bypassPermissions`는 기본 금지한다.
- Researcher 절차는 하나의 Research Summary 형식으로 통일한다.
- Failure Handling 전략은 정책과 handoff 구조에 반영되어 있으나, Validator 반복 루프 전체 자동화는 후속 작업이다.

## 3. Generator wrapper 동작

기본 command shape:

```text
claude --bare --print --model best --effort xhigh --input-format text --output-format json --no-session-persistence --permission-mode auto
```

자동 처리:

- `generator-handoff.v1` JSON 입력 검증
- `--continue`, `--resume`, permission bypass 계열 차단
- `generator-result.json`, `generator-stderr.log`, `generator-run.json` 저장
- 실행 직전 `INSTRUCTION_SENT` 원장 기록
- 실행 결과에 따라 `AGENT_RESULT_RECEIVED` 또는 `RESOURCE_FAILURE` 원장 기록
- `--dry-run`으로 산출물 변경 없이 command shape 확인

## 5. 영향도 및 리스크

제품 API, UI, DB, 배포 설정은 변경하지 않았다.

운영 영향은 다음과 같다.

- Generator 호출이 수동 명령에서 재현 가능한 wrapper 실행으로 바뀌었다.
- 모델·effort·권한 모드가 명시되어 CLI 기본값 의존이 줄었다.
- Researcher 기록 절차가 통일되어 조사 누락을 줄인다.
- Failure Handling 전략의 자동화 범위와 미자동화 범위가 문서로 분리됐다.

남은 리스크:

- Validator-A/B 실행 wrapper는 아직 없다.
- Validator 결과를 읽어 `generator-input-retry-{N}.json`을 자동 생성하지는 않는다.
- 동일 오류 2회 반복 감지와 Conflict Report 작성은 아직 운영자/Analyst 절차다.
