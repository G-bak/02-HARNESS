- **결론:** `git-branch-policy.md`의 Post-merge Branch Cleanup 절을 squash 트리 동등성 기준으로 격상하고, `scripts/clean-merged-task-branches.mjs`를 추가했다. 잔존 4건을 안전 삭제하고 1건은 정책에 따라 보존 처리했다.

- **검증:** `npm run audit:harness`, `npm run check:encoding`, 스크립트 dry-run + 실제 실행 모두 PASS
- **남은 리스크:** `task/TASK-20260427-092` 1건은 트리 차이로 자동 정리 보류 — 별도 수동 검토 필요. 알고리즘은 squash 후 main에 같은 task_id 후속 commit이 추가된 케이스에 대해서만 보장하므로, rebase·cherry-pick 워크플로우에 대한 검증은 후속 과제.
- **조치 필요:** 사용자 승인 후 commit → main squash 머지 → push → 자기 브랜치 정리.

## 사용자 질문 답변

> "git-branch-policy.md 정책상 git branch --merged main에 표시되는 것만 삭제하라는게 맞는거야? 그냥 잔존 브랜치 다 삭제하라는건 좀 위험한가? main에 잘 반영이 됐으면 삭제해도 되는거 아니야?"

1. 직관이 맞다. 기존 정책은 git의 기본 안전장치(`--merged`)에 의존했는데, 이 검사는 일반 merge용으로 설계됐고 squash merge 환경에서는 항상 false negative다.
2. 그렇다고 task_id 문자열 매칭만으로 일괄 삭제하면 false positive 위험이 있다 (squash 후 브랜치에 추가 커밋 누락, 무관 task와 메시지 충돌 등).
3. 정확한 검사는 **브랜치 tip의 tree 객체와 main의 첫 번째 squash 커밋의 tree 객체를 직접 비교**하는 것이다. tree object는 파일 내용의 해시이므로 이름 매칭의 모호성을 제거한다. 이 정책으로 갱신했다.

- `scripts/clean-merged-task-branches.mjs` (신규, 약 220줄)
  - 기본 dry-run, `--force`로 실제 삭제

  - 알고리즘: branch tip tree == 첫 번째 `[TASK-ID]` squash 커밋 tree → safe-delete
