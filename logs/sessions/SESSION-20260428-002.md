> 추천은 B — 같은 문제가 매주 반복될 거라서 정책+스크립트를 동시에 잡는 게 효율적이에요.

> 진행하겠습니다. 먼저 신규 Task 등록 + 브랜치 + 정책 갱신 + 스크립트 작성 + 5건 정리 + 보고서까지 일괄 진행하고, push·머지 직전에 한 번만 사용자 확인 받겠습니다.

## 결정 사항

- 본 작업은 Tier 2로 분류한다. 권위 문서(`docs/operations/git-branch-policy.md`) 변경 + 신규 자동화 스크립트(`scripts/clean-merged-task-branches.mjs`) 추가 + 잔존 브랜치 5건 정리. 제품 런타임 변경 없음.
- 정책 변경의 본질: `git branch --merged main`은 일반 merge용 ancestor 검사라 squash merge 워크플로우에서는 항상 false negative. 트리 동등성(`git diff main..<branch> --quiet`) 검사는 "branch tip의 변경이 main에 모두 반영되었는가"를 정확히 본다.
- task_id 문자열 매칭은 false positive 위험이 있어 1차 안전장치로 채택하지 않는다. 트리 비교가 단일 진실 원천.
- 스크립트 기본은 dry-run. 삭제는 `--force` 명시 필요. 현재 체크아웃된 브랜치와 `main`은 어떤 옵션 조합으로도 삭제 금지.
- 잔존 5건은 사전 수동 확인으로 모두 main에 task_id squash 커밋이 있음을 확인했고, 새 스크립트의 트리 동등성 검사로 한 번 더 검증한 뒤 삭제한다.
- 자기 브랜치(task/TASK-20260428-006) 자체도 main 머지·push 후 같은 스크립트로 정리한다.
