# Tier Classification — 복잡도 분류 기준

**버전:** 1.3 | **최종 수정:** 2026-04-26

---

## 핵심 판단 원칙

> "틀렸을 때 얼마나 빨리, 얼마나 안전하게 되돌릴 수 있는가?"

단순해 보여도 되돌리기 어려우면 Tier 3이다.  
복잡해 보여도 되돌리기 쉬우면 Tier 2다.

---

## Tier 기준표

### Tier 1 — Simple

**"틀려도 1분 안에 되돌린다"**

| 조건 | 기준 |
|---|---|
| 영향 파일 | 1개 이하 |
| 되돌리기 | 즉시 가능 |
| 사이드 이펙트 | 없음 |
| 실패 결과 | 다른 기능에 영향 없음 |

**흐름:** Analyst → Generator → Analyst 자체 검토 → 완료  
**Validator 없음, Researcher 없음**

예시:
- README 오타 수정
- 로그인 버튼 텍스트 변경
- UI 색상 수정
- 주석 추가

---

### Tier 2 — Standard

**"틀리면 고쳐야 하지만 피해는 이 기능에 국한된다"**

| 조건 | 기준 |
|---|---|
| 영향 파일 | 복수 |
| 되돌리기 | 수정 사이클로 해결 가능 |
| 사이드 이펙트 | 해당 기능 범위 내 |
| 실패 결과 | 다른 기능은 정상 동작 |

**흐름:** Analyst → (Researcher?) → Generator → Validator-A (Codex) → PASS → main 머지

예시:
- 사용자 프로필 페이지 신규 개발
- 결제 모듈 연동
- 새 필터 기능 추가
- API 응답 형식 변경 (내부 사용)

---

### Tier 3 — Critical

**"틀리면 되돌리기 어렵거나 다른 사람 데이터에 영향을 준다"**

**해당 조건 (하나라도 해당하면 즉시 Tier 3):**

```
[ ] 보안 또는 인증 로직 변경
[ ] DB 스키마 변경 또는 데이터 마이그레이션
[ ] 외부 공개 API의 breaking change
    (응답 필드 제거·타입 변경·엔드포인트 폐기·인증 방식 변경·데이터 계약 변경)
    ※ non-breaking 추가(필드 추가, 메시지 수정)는 Tier 2로 처리
[ ] 프로덕션 환경 설정·시크릿·데이터·라우팅에 즉시 반영되는 작업
[ ] 실패 시 데이터 손실 또는 서비스 중단 가능성
```

프로덕션 관련 판단:

| 작업 | Tier |
|---|---|
| 프로덕션 시크릿, DB, 라우팅, 인증, 결제, 데이터 처리 변경 | Tier 3 |
| 이미 검증된 커밋을 배포만 수행 | Tier 2 또는 별도 운영 절차 |
| 정적 문구/스타일 변경이 배포에 포함됨 | 변경 범위 기준 Tier 1~2, 단 배포 실패 시 서비스 중단 가능하면 Tier 3 |
| main 머지 자체 | git-branch-policy.md의 머지 조건을 따름 |

**흐름:** Analyst → (Researcher?) → Generator → Validator-A + Validator-B (독립 병렬) → 둘 다 PASS → Analyst 최종 승인 → main 머지

예시:
- 로그인 세션 토큰 처리 방식 변경
- users 테이블 컬럼 삭제
- OAuth 연동
- 프로덕션 환경 설정 변경
- 외부 결제 API 연동

## 판단 기준 비교표

| 항목 | Tier 1 | Tier 2 | Tier 3 |
|---|---|---|---|
| 영향 파일 수 | 1개 이하 | 복수 | 시스템 전체 가능 |
| 되돌리기 | 즉시 | 수정 사이클 | 어렵거나 불가능 |
| 타 기능 영향 | 없음 | 없음 | 있음 |
| 데이터 위험 | 없음 | 없음 | 있음 |
| 보안 관련 | 없음 | 없음 | 있음 |
| Validator | 없음 | Validator-A | A + B |

## 작업 중 Tier 재분류 (Runtime Reclassification)

Tier는 Task Spec 수립 시 분류되지만, 작업 중 발견으로 상향 재분류될 수 있다.  
**하향 재분류는 일반적으로 허용하지 않는다.**

### 하향 재분류 예외 (엄격 제한)

**하향 재분류는 원칙적으로 허용하지 않는다.**  
단, 아래 조건을 **모두** 충족하는 경우에만 사용자 명시적 승인 하에 허용한다:

```
[ ] 사용자가 직접 하향 재분류를 요청했음
[ ] Analyst가 하향 근거를 Task Spec에 문서화했음
[ ] 하향 후 Tier의 Validator 구성이 위험을 충분히 커버함을 Analyst가 확인했음
[ ] 하향 사실과 근거를 보고서에 명시했음
```

조건 중 하나라도 미충족 시 하향 재분류를 진행하지 않는다.

---

### 재분류 트리거

| 발견 주체 | 조건 | 즉시 조치 |
|---|---|---|
| Generator | 구현 중 Tier 3 조건 발견 | 작업 즉시 중단 → Analyst에게 재분류 요청 보고 |
| Validator | 검증 중 결과물에서 Tier 3 조건 발견 | FAIL 처리 대신 Tier 재분류 요청 Analyst에게 전달 |
| Analyst | 에이전트 보고 수신 또는 직접 발견 | Task Spec `complexity_tier` + `tier_rationale` 갱신 |

### 재분류 후 처리

Tier 2 → Tier 3 전환 시:
1. Analyst가 `complexity_tier`와 `tier_rationale`을 갱신한다
2. Validator-B (Gemini CLI)를 즉시 추가 투입한다
3. Generator가 이미 결과물을 생성한 경우에도 Validator-B 검증을 생략하지 않는다
4. 재분류 사실과 이유를 보고서에 기록한다

## 예시 분류 케이스

| 작업 | 처음 보이는 Tier | 실제 Tier | 이유 |
|---|---|---|---|
| 로그인 버튼 텍스트 변경 | 1 | 1 | 1파일, 즉시 되돌리기 가능 |
| 프로필 페이지 신규 개발 | 2 | 2 | 3파일, 다른 기능 영향 없음 |
| "간단한" 비밀번호 정책 수정 | 1 | 3 | 인증 로직 변경 → 즉시 Tier 3 |
| 내부 유틸 함수 리팩터링 | 2 | 3 | 해당 함수가 인증 미들웨어에서 사용 중 → Tier 3 |
| 세션 토큰 저장 방식 변경 | 3 | 3 | 명백한 Tier 3 |
